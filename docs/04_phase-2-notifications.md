# See-Sweet Phase 2 — Notification System

## Plan

### Goal

Build a real-time, LLM-generated notification system that replaces static pre-written messages with dynamic, contextually aware content assembled by the assistant. The system is designed to be extended over time to support additional notification categories, delivery channels, and role-based sources (user-facing, system-generated, admin-broadcast).

---

### Guiding Principles

- **No pre-written copy.** Every notification that involves a message body is generated by the LLM at trigger time, using user context (name, locale, preferences) to produce something that feels personal and relevant, not templated.
- **Persistent.** Notifications are stored in the database, scoped to the user. They can be read, dismissed, or kept for reference. This is not a toast queue — it is a lightweight inbox.
- **Real-time.** Delivery to the client uses Server-Sent Events (SSE), consistent with the existing streaming pattern used in chat. No WebSocket dependency is introduced.
- **Typed and extensible.** Each notification carries a `category` and `source` field, making it straightforward to add new categories without schema migrations.
- **Localization-aware.** The LLM prompt used to generate notification text includes the user's locale, so the generated content naturally follows the user's language preference.

---

### Notification Shape

Each notification record captures the following:

| Field       | Type       | Description                                                                             |
| ----------- | ---------- | --------------------------------------------------------------------------------------- |
| `id`        | `cuid()`   | Unique identifier.                                                                      |
| `userId`    | `String`   | Owner. All reads are scoped to this field.                                              |
| `category`  | `String`   | Machine-readable type: `welcome_signup`, `welcome_login`, `system`, `token_limit`, etc. |
| `source`    | `String`   | Who produced it: `assistant` (LLM-generated) or `system` (platform-generated).          |
| `title`     | `String`   | Short heading (LLM-generated when source is `assistant`).                               |
| `body`      | `String`   | Full message body.                                                                      |
| `read`      | `Boolean`  | Whether the user has opened or acknowledged it. Defaults `false`.                       |
| `dismissed` | `Boolean`  | Soft-delete flag; hides from the panel without removing the record.                     |
| `metadata`  | `Json?`    | Reserved for future structured payloads (e.g., token counts, event refs).               |
| `createdAt` | `DateTime` | Record creation timestamp.                                                              |

---

### Categories (Initial + Planned)

| Category         | Source      | Trigger                                    | Status    |
| ---------------- | ----------- | ------------------------------------------ | --------- |
| `welcome_signup` | `assistant` | User completes sign-up                     | In scope  |
| `welcome_login`  | `assistant` | User signs in to an existing account       | In scope  |
| `system`         | `system`    | Platform-level events (maintenance, etc.)  | Stub only |
| `token_limit`    | `system`    | Monthly token quota approaching or reached | Planned   |

The `category` field is a plain string, not an enum, so new categories can be added at the application layer without a schema migration.

---

### LLM Generation

Notifications sourced from `assistant` are generated by the same local Ollama instance used for chat (`lib/chat/ollama.ts`). A dedicated generation function lives in `lib/notifications/generate.logic.ts`.

The prompt includes:

- User name and locale.
- The notification category (to shape tone and content).
- A brief instruction set: be warm, concise, personal — no generic boilerplate.
- No conversation history (notifications are standalone messages, not chat turns).

The `think` flag is set to `false` — reasoning overhead is unnecessary for short notification copy.

Generation is non-blocking at auth time: the auth event handler triggers generation and insertion as a background task using `void` fire-and-forget, so it does not delay the sign-up or login response.

---

### Real-Time Delivery

The client subscribes to a `GET /api/notifications/stream` SSE endpoint after authentication. The endpoint:

1. Authenticates the request (session cookie).
2. Opens a `ReadableStream` and sends any undelivered notifications as `data:` events.
3. Keeps the connection alive with a periodic comment ping (`": keep-alive\n\n"`).
4. On new notification insertion, the SSE stream pushes the payload to the connected client.

For the initial scope, push from the server to the stream is achieved with a lightweight in-memory emitter (a `Map` of `userId → Set<controller>`) held at module scope in the SSE route. This is sufficient for a single-instance local deployment. A notes entry documents that for multi-instance deployments, this should be replaced with a Redis pub/sub or similar broadcast mechanism.

---

### Client-Side Architecture

A `useNotifications` hook (`hooks/use-notifications.ts`) manages the SSE connection, incoming notification state, and read/dismiss mutations. The hook:

- Establishes the SSE connection on mount.
- Appends incoming events to a local array.
- Exposes `markRead(id)` and `dismiss(id)` methods, which call `PATCH /api/notifications/[id]`.
- Exposes an `unreadCount` derived value for the badge.

The notification bell icon lives in `components/shared/site-header.tsx` and shows a badge when `unreadCount > 0`. Clicking it opens a `NotificationPanel` (a `Sheet` or `Popover`) listing recent notifications with inline dismiss controls.

---

### API Routes

| Method  | Path                          | Description                                             |
| ------- | ----------------------------- | ------------------------------------------------------- |
| `GET`   | `/api/notifications`          | List notifications for the current user (paginated).    |
| `GET`   | `/api/notifications/stream`   | SSE endpoint; pushes new notifications in real time.    |
| `PATCH` | `/api/notifications/[id]`     | Mark as read and/or dismissed.                          |
| `POST`  | `/api/notifications/generate` | Internal trigger; called server-side after auth events. |

The `POST /api/notifications/generate` route is not user-callable. It is triggered from inside auth event hooks or server actions, and is protected by a shared secret header check (never exposed to the client route tree directly).

---

### Auth Integration Points

Better Auth exposes `databaseHooks` where custom logic can be attached to user lifecycle events. Two hooks are registered:

- `user.create.after` → triggers `welcome_signup` notification generation.
- `session.create.after` → triggers `welcome_login` notification generation (skipped if the session was created within the same minute as `user.create`, to avoid sending both on the same sign-up flow).

### Prisma Schema Addition

```prisma
model Notification {
  id          String   @id @default(cuid())
  userId      String
  category    String
  source      String   @default("assistant")
  title       String
  body        String
  read        Boolean  @default(false)
  dismissed   Boolean  @default(false)
  metadata    Json?
  createdAt   DateTime @default(now())

  user User @relation(fields: [userId], references: [id], onDelete: Cascade)

  @@index([userId, dismissed, createdAt(sort: Desc)])
  @@map("notification")
}
```

### Scalability Notes

- Adding a new notification category requires: a trigger site, a prompt variant in `generate.logic.ts`, and an optional icon/style mapping in `NotificationItem.tsx`. No schema changes.
- Adding a new `source` (e.g., `admin`) requires: a new trigger path and a source guard in the API. No schema changes.
- Replacing the in-memory emitter with Redis pub/sub for multi-instance: swap only `lib/notifications/emitter.ts`.
- Future: batch digests (daily briefing notification), scheduled notifications, and priority levels can all be added as new `category` values and `metadata` fields.
